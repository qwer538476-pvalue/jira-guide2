Part 1
Jira(지라) 핵심 개념
복잡한 협업 도구를 올바르게 활용하기 위해, 핵심 구성 요소를 우리 조직의 실제 업무 흐름에 빗대어 정의합니다
🏬
거대한 물류 창고

스페이스 (프로젝트)

Jira(지라)에서 이슈들을 저장하는 저장소 역할. 모든 이슈가 모이는 하나의 공간입니다.

📝
요청 정의서

이슈 (작업·프로세스·요청·기능)

과제 요청 시 발행되는 하나의 업무 단위. 해당 이슈에 대한 모든 정보와 설명이 이곳에 담깁니다. 각 타입에 대한 정의는 자유롭게 설정하여 사용합니다.

✅
개인별 체크리스트

하위태스크 (Sub-task)

기획·디자인·프론트엔드·백엔드 등 파트별 실제 작업 항목으로 나누어 각각 담당자를 지정하는 단위. 부모 이슈 안에 생성되는 '개인별 작업리스트'입니다.

실제 작업자를 지정합니다. 담당자가 모호한 경우, 이승철 부장으로 우선 지정합니다.
📌 기타 원칙: 이슈 생성 시 제목과 내용은 최대한 상세히 정의합니다. 이슈의 기한은 최초 리뷰(킥오프) 후 협의하여 결정하고, 하위 작업은 담당자가 전체 기한에 맞춰 논의 후 결정합니다. 업무 연관성이 있는 이슈끼리는 서로 연결해두는 기능(디펜던시)을 활용해 관련 이슈를 한눈에 볼 수 있습니다.
Part 3
하위태스크 운용 및 커뮤니케이션 규칙
정보의 파편화를 막기 위해 '부모 티켓에 모든 정보 입력, 하위태스크는 일정 관리'의 원칙을 준수합니다
하위태스크 생성
기획·디자인·프론트엔드·백엔드·QA 등 각 파트별 작업 단위로 생성하고, 실제 작업자를 담당자로 지정합니다.
커뮤니케이션 일원화
작업 내용 수정, 논의 사항, 의사결정 등 모든 기록은 반드시 부모 티켓의 코멘트에 작성합니다.
⛔ 절대 준수 사항 — 하위태스크 운영 금지 규칙

하위태스크 내부에 작업 내용을 상세히 기재하거나 코멘트를 작성하는 행위를 엄격히 금지합니다. 모든 정보는 부모 티켓에 집중되어야 합니다.

❌ 하위태스크에서 금지

작업 상세 내용 기재 · 논의 코멘트 작성 · 의사결정 기록

Critical 진행중
1 — 이슈 개요 및 목적
3/27 React 전환 이후 회원 관련 이벤트명이 잘못 수집되고 있어, 정상 이벤트 기준으로 데이터를 복구하기 위함입니다.
2 — 상세 작업 내용
로그인·회원가입·회원탈퇴 이벤트가 login1 · sign_up1 · sign_out1으로 잘못 수집되어, 정상 이벤트(login2 등)는 사실상 미수집 상태입니다.
로그아웃(logout2 · logout1)은 이름 오류가 아닌 이벤트 자체가 발생하지 않는 문제입니다.
  • GTM 트리거 조건을 정상 이벤트명 기준으로 수정
  • 로그아웃 이벤트 발생 여부 코드 레벨에서 확인
3 — 완료 조건

Part 7
전략실 확인 요청 사항
교육 자리에서 개발팀에 직접 요청한 3가지 사항 — 개발팀이 확인 후 메일로 회신 예정
1
전략실 내부에 팀 만들기 권한 가능 여부
이유
전략실 내부에 여러 하위 조직(팀)이 있어, 업무 영역별로 구분해서 관리할 필요가 있는데 현재는 전략실 전체가 하나의 공간으로 통합되어 있어 분리가 필요합니다.
현재상태
교육 중 즉답이 어려워, 개발팀이 권한 확인 후 회신하기로 함
회신 대기 중
2
티켓 삭제 권한을 전략실 팀장 단위로만 부여 가능한지
이유
불필요하거나 잘못 생성된 테스트성 티켓을 정리할 권한이 필요한데, 전원에게 권한을 부여하면 오남용 우려가 있어 팀장 단위로 제한된 권한 부여를 원합니다.
현재상태
개발팀이 가능 여부 확인 후 회신하기로 함
회신 대기 중
3
할일→진행중→검토중→완료 상태 변경 권한을 팀 전원이 가지는 게 맞는지
이유
요청자가 아닌 사람이 임의로 상태를 변경하거나, 작업 중인 티켓이 실수로 되돌려지는 등의 혼선을 막기 위해 권한 범위를 명확히 하고자 합니다.
1차의견
권한을 세분화하는 기능 자체가 Jira(지라)에 없는 것으로 확인됨 → 일단 "담당자만 변경" 룰을 권장하나, 이는 확정 답변이 아니며 추가 확인 후 정식 회신 예정
회신 대기 중
